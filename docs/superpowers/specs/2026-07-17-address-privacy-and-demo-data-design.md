# 地址分层、微信校验与演示数据替换设计

## 目标

修复演示订单中文乱码和关键字段缺失问题，将地址拆分为粗略地址与详细地址，并确保教师端只能看到粗略地址。删除批次 `58f88b6d-818e-410d-835b-9dfb1c4860eb` 产生的 100 条错误演示订单，重新生成并发布 50 条内容完整、格式正常的虚拟订单。

## 数据模型

- 在 `orders` 表新增 `rough_address` 字段，表示教师可见的通勤范围，例如“交大创新港附近”或“灞桥区纺织城林河春天”。
- 现有 `address` 字段继续表示详细地址，例如小区楼栋、单元或门牌，仅供中介端查看。
- `area` 继续表示行政区域，用于筛选，不与粗略地址混用。
- 家长微信 `parent_wechat` 是新建、编辑和导入发布订单的必填字段。
- 发布订单的中介账号必须有非空中介微信；管理员账号不用于发布订单。
- 为兼容历史数据，数据库约束允许旧记录暂时存在空字段，但所有新增或更新订单必须满足新规则。

## 自动识别与导入

- “粗略地址”“地址”“位置”“补习地址”识别为 `roughAddress`。
- “详细地址”“具体地址”“门牌地址”识别为 `address`。
- 旧格式中只有“地址”时，将其视为粗略地址，不猜测详细门牌。
- 粗略地址、详细地址或家长微信任一缺失时，导入记录进入待审核，不能直接发布。
- 表格导入增加粗略地址和详细地址的独立列映射。
- 重复检测仍以家长联系方式以及详细地址、年级、科目组合为依据。

## 权限与页面展示

### 教师端

- API 只返回粗略地址，不返回详细地址、家长微信、家长电话、内部备注或原始文本。
- 订单卡片和详情将粗略地址显示为“地址”。
- 中介微信在订单详情中明确显示，方便教师联系中介。

### 中介端

- 录入、识别复核、待审核和订单详情分别展示粗略地址与详细地址。
- 中介可查看家长微信；电话保持选填。
- 缺少家长微信或发布账号缺少中介微信时，页面和服务端都拒绝发布并给出明确提示。

## 演示数据替换

- 仅删除批次 `58f88b6d-818e-410d-835b-9dfb1c4860eb` 关联的 100 条乱码订单、导入项和批次记录，不修改原有 49 条数据。
- 新建 50 条西安地区虚拟家教订单，字段包括年级、科目、成绩、时间、频率、时长、报价、教师要求、粗略地址、详细地址和家长微信。
- 家长微信使用格式有效且唯一的虚拟账号；订单绑定到已有且已填写微信的中介账号。
- 虚拟标记只存放在中介可见的内部备注中，不破坏演示页面的正常订单观感。
- 生成和写入过程使用 UTF-8 文件或程序内部 Unicode 数据，不再通过可能损坏中文的 PowerShell 文本管道传输。

## 数据迁移与回滚

- 新增顺序迁移 `009`，创建 `rough_address` 字段及必要约束。
- 迁移先用现有 `area` 为历史订单回填粗略地址；不从旧详细地址自动截取内容，避免误把楼栋门牌暴露给教师端。
- 删除错误批次前先按批次 ID 核对订单数必须为 100；数量不符时停止，不执行删除。
- 新批次发布后核对恰好 50 条已发布订单、0 条待审核，并验证教师接口不含敏感字段。

## 测试与验收

- 解析测试覆盖粗略地址、详细地址、家长微信以及用户提供的示例格式。
- 导入校验测试覆盖三个必填字段缺失时进入待审核。
- 仓储和 API 测试证明教师端只得到粗略地址，中介端得到两个地址和家长微信。
- 写入测试证明家长微信和中介微信缺失时无法发布。
- 完整测试套件通过后运行数据库迁移、定向删除和 50 条数据导入。
- 最终核对数据库总订单数为 99 条：原有 49 条加新生成 50 条。
- 将代码、迁移、测试和数据生成脚本推送到 GitHub `main`，并确认部署检查通过。
